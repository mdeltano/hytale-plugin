Пользовательский интерфейс можно изменить с помощью **серверного** моддинга. Многие страницы пользовательского интерфейса, показываемые игрокам, могут быть изменены владельцами серверов с помощью модов.

## Кастомизация пользовательского интерфейса в Hytale

Hytale предоставляет обширные возможности настройки пользовательского интерфейса посредством моддинга на стороне сервера, что позволяет владельцам серверов изменять многие страницы пользовательского интерфейса, с которыми взаимодействуют игроки.

### Управление потоками

Система пользовательского интерфейса в Hytale соответствует определенной схеме управления потоком:

1. Сервер отправляет пользовательские данные клиенту
2. Клиент рендерит пользовательский интерфейс
3. Пользователь взаимодействует с интерфейсом (например, нажатие кнопки)
4. Клиент отправляет событие взаимодействия на сервер
5. Сервер выполняет логику обратного вызова для события

Эта архитектура означает:

* Клиент отвечает за рендеринг пользовательского интерфейса и отправку взаимодействия пользователя на сервер
* Сервер обрабатывает всю фактическую логику взаимодействия с пользователем.

<Callout type="warning" title="Возможные последствия для производительности">
  <span>
    Поскольку сервер обрабатывает всю логику пользовательского интерфейса, существует неизбежная задержка, которая влияет на скорость отклика пользовательского интерфейса. Нажатия кнопок и другие взаимодействия должны
    дойти до сервера, прежде чем вступить в силу, а это означает, что сетевой пинг напрямую
    влияет на отзывчивость пользовательского интерфейса.
  </span>

  <span>
    Кроме того, поскольку моддинг на стороне клиента не поддерживается, модифицированные значения не сохраняются на клиенте.
  </span>
</Callout>

### UI Layout Assets

Система настройки интерфейса Hytale работает по следующей структуре:

* **Макет пользовательского интерфейса**: Отправляется клиенту в качестве ресурса.
* **Поведение пользовательского интерфейса**: Определено в коде Java
* **Пользовательские макеты**: Включает создание существующих макетов интерфейса, а не переназначение существующих (в отличие от модификаций инвентаря в Minecraft)

### UI Framework Implementations

В настоящее время Hytale поддерживает несколько пользовательских реализаций с планами консолидации на NoesisGUI:

#### Legacy реализация

* Служит примером возможностей моддинга сервера.
* Использует комбинации из внутриигровых элементов, разметки ассетов пользовательского интерфейса и ко Java модов.
* Использует legacy язык разметки
* Будет устаревшее в пользу NoesisGUI

#### NoesisGUI (первичный фреймворк)

* Использует язык разметки XAML для определения интерфейса
* Поддержка нескольких языков для интернационализации
* Выполняется на C# привязках с клиентской стороны (не C++)
* Представляет собой будущее направление интерфейса Hytale's

#### Third Framework

* Сейчас на стадии завершения использования
* Будет заменено NoesisGUI в рамках консолидации

### Особенности производительности

Поскольку обработка пользовательского интерфейса выполняется на стороне сервера, необходимо учитывать некоторые характеристики производительности:

* **Влияние на задержку**: нажатия кнопок и взаимодействия с пользовательским интерфейсом должны быть отправлены на сервер, прежде чем вступят в силу, а это означает, что сетевой пинг напрямую влияет на скорость отклика пользовательского интерфейса.
* **Нет сохранения на стороне клиента**: поскольку модификация на стороне клиента не поддерживается, измененные значения не сохраняются на стороне клиента.
* **Оптимизированная архитектура**: система Hytale разработана для минимизации задержек за счет эффективных протоколов связи и оптимизированной производительности сервера, что гарантирует бесперебойное взаимодействие, несмотря на требования к обработке на стороне сервера.

Архитектура приоритизирует безопасность и согласованность путем централизации логики пользовательского интерфейса на сервере, сохраняя отзывчивый опыт пользователя
